## 🟠 ÉLEVÉ

### 🟠 H-1 — AES-CBC au lieu d'AEAD (AES-GCM / ChaCha20-Poly1305)
**Fichiers** :
- `lib/services/vault_service.dart:483, 519` (AESMode.cbc)
- `lib/services/import_export_service.dart:266, 324` (AESMode.cbc)
- `lib/services/heritage_service.dart:239, 285` (AESMode.cbc)

**Description** : tout le vault, le backup `.ptbak` et le snapshot héritier utilisent AES-256-CBC avec **MAC séparé HMAC-SHA256** (Encrypt-then-MAC). C'est correct cryptographiquement (la version v3 inclut AAD = `version|iter|salt`), mais :
- CBC nécessite un padding (PKCS#7) → vulnérable aux *padding oracles* si jamais le code de vérification MAC venait à être contourné dans un futur refactor (régression facile).
- AES-GCM ou ChaCha20-Poly1305 fournit AEAD natif (atomique, plus difficile à mal utiliser) et a une réputation bien meilleure pour des nouveaux designs en 2026.

**Impact** : exposition latente à un padding oracle si la vérification MAC est un jour cassée par mégarde (régression). Image marketing : "Pack 5/5" sans AEAD est un argument que les reviewers techniques pourront contester.

**Correctif** :
1. Ajouter une **migration v4** : à la prochaine ouverture, ré-chiffrer en AES-256-GCM (`pointycastle` ou `cryptography` package) avec un nonce 12 octets unique par save.
2. Conserver le code v1/v2/v3 read-only pour migrer les vaults existants.
3. Lever le flag `version: 4` dans le JSON ; supprimer le HMAC séparé (intégré au tag GCM).
4. Mêmes changements pour `.ptbak` (v3) et `pt_heir.enc` (v2).

### 🟠 H-2 — KDF : PBKDF2 plutôt qu'Argon2id
**Fichiers** :
- `lib/services/vault_service.dart:16-46, 107` (`pbkdf2Worker` + `_currentIterations = 600000`)
- `lib/services/import_export_service.dart:254`
- `lib/services/heritage_service.dart:49`

**Description** : PBKDF2-HMAC-SHA256 600 000 itérations est conforme OWASP 2023, mais **Argon2id** est l'état de l'art (résiste GPU/ASIC/TMTO) et standard de facto pour les password managers modernes (Bitwarden, 1Password, KeePassXC). PBKDF2 reste calculable très efficacement sur GPU.

**Impact** : sur un master password humain (~40-50 bits d'entropie), un attaquant avec rig GPU peut tester ~10⁵-10⁶ passwords/s contre PBKDF2-SHA256, vs ~10²-10³ avec Argon2id (params standards). Différence de plusieurs ordres de grandeur sur le coût d'une attaque hors-ligne après vol du fichier `pt_vault.enc`.

**Correctif** :
1. Migrer vers Argon2id via `argon2` (dart) ou un wrapper natif. Paramètres recommandés OWASP 2024 : `m=19 MiB, t=2, p=1` (mémoire-hard) — ou `m=46 MiB, t=1, p=1`.
2. Ajouter un champ `kdf: "argon2id"` au format vault (avec params `m`, `t`, `p`). Maintenir lecture du legacy `kdf: "pbkdf2"` pour migration.
3. À la 1ère unlock après MAJ, re-dériver avec Argon2id et ré-écrire le vault.

### 🟠 H-3 — Master key dérivée NON liée au Keystore Android
**Fichier** : `lib/services/vault_service.dart` (toute la classe).

**Description** : la clé dérivée du master password reste **purement applicative**. Sur un device rooté ou avec `frida-server` attaché, un attaquant peut :
- lire `pt_vault.enc` + `pt_salt` directement,
- attaquer le master password offline (cf. H-2),
- ou hooker la mémoire pour récupérer `_key` après unlock.

La clé biométrique stockée dans `biometric_storage` (AndroidX Security/Keystore) est elle bien protégée. Mais elle ne contient **que** la clé déjà dérivée — elle ne renforce pas la dérivation initiale.

**Impact** : la promesse "Pack 5/5" implique que le vault chiffré seul ne soit pas crackable raisonnablement. Sans Keystore (HSM/StrongBox) qui force un compute on-device limité par hardware, l'attaquant offline n'a aucune limite de débit.

**Correctif** : envisager un schéma "two-secret key derivation" :
1. Créer une clé Keystore non-extractible (StrongBox si dispo) à l'install via `flutter_secure_storage` v10 (déjà dépendance) ou un MethodChannel direct sur AndroidKeyStore.
2. La clé finale du vault = `HKDF(keystore_unwrap(salt) || pbkdf2/argon2(masterPassword, salt))`.
3. Conséquence : impossible d'attaquer offline le vault même si l'attaquant a le fichier (il lui manque le secret hardware-bound).

C'est l'approche utilisée par 1Password (Secret Key) et constitue un saut qualitatif réel.

### 🟠 H-4 — `FlutterFragmentActivity` exporté + alias multiples
**Fichier** : `android/app/src/main/AndroidManifest.xml:21-63`

**Description** : `MainActivity` est `android:exported="true"` (requis Android 12+ pour activity launcher). Pas de filtre intent-filter explicite sur l'activity → seuls les alias l'invoquent. Mais l'alias `MainAliasDecoy` (Calculatrice) reste un point de contact externe. Le `taskAffinity=""` est bien posé, mais aucun `singleInstance` / `excludeFromRecents`.

**Impact** : limite — pas d'`onNewIntent` traitant un payload sensible. Mais une app malveillante peut lancer `Calculatrice` pour ouvrir Pass Tech volontairement et observer les transitions (timing).

**Correctif** :
- Ajouter `android:excludeFromRecents="true"` sur `MainActivity` pour éviter même la miniature dans Recents (FLAG_SECURE le couvre déjà visuellement, mais le titre reste visible).
- Vérifier qu'aucun deep link n'est défini (audit OK : aucun).
- Documenter dans `SECURITY.md` que les alias sont des entry points UI uniquement.

### 🟠 H-5 — Format JSON non chiffré exporté via `Share.shareXFiles`
**Fichier** : `lib/screens/settings_screen.dart:545-554` (`_exportVault`)

**Description** : `_exportVault()` appelle `VaultService().exportJson()` (entries en clair) → écrit `pass_tech_export.json` dans le **temporary directory** → partagé via `share_plus`. Aucun avertissement bloquant, juste un bouton dans Settings.

**Impact** : le fichier reste dans `/data/data/.../cache/` ou sandbox temporaire jusqu'au prochain GC OS. Un attaquant local (ou backup utilisateur déclenché involontairement) le récupère en clair. Le mode "Sécurité radicale" doit dissuader.

**Correctif** :
1. Demander une confirmation explicite **avec champs warning rouge** ("export non chiffré : un attaquant qui accède à ce fichier verra TOUS vos mots de passe").
2. Après `Share.shareXFiles`, **supprimer immédiatement** le fichier (`file.delete()`).
3. Mieux : retirer purement l'option `_exportVault` (laisser uniquement `_exportEncrypted` `.ptbak`) ou cacher derrière un toggle "mode développeur".

## 🟡 MOYEN

### 🟡 M-1 — `String.fromCharCodes(bytes)` sur import
**Fichier** : `lib/screens/settings_screen.dart:610`

Décodage octets → string sans `utf8.decode`. Casse les caractères accentués des CSV/JSON exportés ailleurs. Impact : entries importées corrompues silencieusement.
**Correctif** : `String content = utf8.decode(bytes, allowMalformed: true);`

### 🟡 M-2 — `_constEq` retourne tôt sur length mismatch
**Fichiers** : `vault_service.dart:578-583`, `import_export_service.dart:342-347`, `heritage_service.dart:227-232`

Le `if (a.length != b.length) return false;` leak la longueur. Pour HMAC-SHA256 c'est toujours 32, donc inoffensif ici, mais **ne pas l'utiliser** pour comparer des secrets de longueur variable. À noter dans un commentaire pour éviter une mauvaise réutilisation future.

### 🟡 M-3 — `_wipeKey()` ne couvre pas les sublist
**Fichier** : `vault_service.dart:481, 517` (`encKey = enc.Key(_key!.sublist(0,32))` puis `macKey = _key!.sublist(32)`).

`sublist` crée une **copie** des octets. Quand `_wipeKey` zéroïse `_key`, les copies restent. Idem dans `heritage_service.dart`. La JVM/Dart conserve donc des sous-buffers de la clé en RAM jusqu'au GC.
**Correctif** : créer des Uint8List.view sur le buffer principal au lieu de `sublist`, ou bien zéroïser les copies dérivées avant de sortir des fonctions `_decryptVault`/`_saveVault`/`_writeSnapshot`.

### 🟡 M-4 — `String _password` du générateur reste en RAM jusqu'à dispose
**Fichier** : `lib/screens/generator_screen.dart:27` + tous les `TextEditingController` dans `setup_screen.dart`, `unlock_screen.dart`.

Les `String` Dart sont immuables → impossible de les wipe activement. `_passCtrl.clear()` ne libère pas la string sous-jacente, le GC s'en occupera *un jour*.
**Correctif** : limiter le scope (variables locales), forcer une re-allocation `pass = '*' * pass.length` puis `clear()`. Documenter l'attaquant model : "des secrets restent dans la heap Dart jusqu'au GC ; mitigations : auto-lock court + FLAG_SECURE bloquent les screenshots de heap dump".

### 🟡 M-5 — `BiometricStorage` valide la session pendant 1 read sans rebind
**Fichier** : `vault_service.dart:91` (`authenticationValidityDurationSeconds: -1`).

`-1` = chaque opération demande une auth → bien. Mais **vérifier** que `biometric_storage` v5 implémente bien `setUserAuthenticationRequired(true)` côté Keystore (il le fait par défaut, OK). Ajouter un test/assertion runtime au démarrage (`canAuthenticate()` + log).

### 🟡 M-6 — Pas de check `setInvalidatedByBiometricEnrollment`
**Fichier** : `vault_service.dart:91-104`

Si l'utilisateur ajoute/retire une empreinte, la clé Keystore liée à la biométrie devrait être **invalidée** (`setInvalidatedByBiometricEnrollment(true)`). `biometric_storage` ne l'expose pas explicitement → vérifier la version 5.0.1 dans le code natif (sinon, écrire un wrapper natif maison).
**Correctif** : ajouter dans la roadmap. Document utilisateur : "le retrait/ajout d'empreinte invalide la déverrouillage biométrique".

### 🟡 M-7 — `_phishingService.lastDomain` + race condition
**Fichier** : `kotlin/.../PhishingDetectorService.kt:38-65`

`@Volatile var _snapshot` est lu côté Dart via MethodChannel. L'AS écrit après chaque WINDOW_*. Si le user copie au moment précis où le navigateur change de page, la vérif peut comparer un domaine périmé. Le `FRESHNESS_MS=60_000L` est généreux ; recommandé de réduire à 5-10 s pour les copies critiques.

### 🟡 M-8 — CSV parser : pas de limite de cellules / DoS
**Fichier** : `import_export_service.dart:203-246`

Le parser custom n'a pas de limite par-cellule (sur un CSV malicieux avec une cellule de 50 Mo en quotes ouvertes, `StringBuffer` croît sans borne). La limite globale `_maxImportBytes = 50 MB` cap l'ensemble, donc l'impact est borné, mais le worst-case mémoire en RAM est doublé (StringBuffer + List) → ~200 MB. Ajouter une borne `if (buf.length > 10*1024*1024) throw ...`.

### 🟡 M-9 — `version` non validé strictement à la lecture
**Fichier** : `vault_service.dart:463` `final version = raw['version'] as int? ?? 1;`

Si un attaquant remplace le fichier par un v9999 inconnu, le code prend la branche `version >= 3`. Pas de risque crypto immédiat (le HMAC échoue), mais un attaquant peut DoS via une `version: 99999` qui déclenche des branches futures. Ajouter `if (version > _currentVersion) return false;`.

### 🟡 M-10 — `flutter_secure_storage` v10 — déprécation EncryptedSharedPreferences
**Fichier** : `pubspec.yaml:15` (`flutter_secure_storage: ^10.0.0`)

Le commentaire l 65-67 du vault explique la migration. Vérifier que **les anciens utilisateurs (1.x → 1.12.6)** ont bien migré : ajouter un test E2E ou un log discret en debug.

### 🟡 M-11 — HIBP / GitHub : User-Agent identifie l'app
**Fichiers** : `breach_service.dart:21` (`'PassTech'`), `app_update.dart` (via `files_tech_core`).

Un attaquant réseau (ISP, MITM TLS si trust-anchors compromises) peut détecter qu'un device utilise Pass Tech. Pas critique (config réseau strictement system-anchors, pas de cleartext), mais le User-Agent doit rester générique. Suggéré : `Mozilla/5.0` ou paramétrable.

### 🟡 M-12 — `enableV1Signing = true`
**Fichier** : `android/app/build.gradle.kts:50`

V1 (jar signing) est connue pour être falsifiable (Janus CVE-2017-13156 sur Android <7). `minSdk` est `flutter.minSdkVersion` (à vérifier — Flutter 3.11+ → 21 par défaut, donc Android 5.0+).
**Correctif** : si `minSdk >= 24`, désactiver `enableV1Signing = false` (V2/V3 suffisent). Si `minSdk < 24`, garder.
